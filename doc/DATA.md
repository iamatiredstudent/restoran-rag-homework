Данные проекта
Назначение данных
Данные используются для учебной RAG-системы по теме ресторанной аналитики. Корпус имитирует базу знаний для ресторана или ресторанной группы, которая хочет управлять закупками, складскими остатками, списаниями, поставщиками, прогнозированием спроса и маржинальностью на основе данных.
Цель датасета — обеспечить работу полного RAG-pipeline: загрузка исходных данных, преобразование записей в документы, нарезка документов на чанки, построение TF-IDF индекса, retrieval, формирование demo-ответа и отображение источников в Streamlit-интерфейсе.
Источник данных
В проекте используется учебный синтетический датасет, созданный специально для выполнения домашнего задания.
Датасет генерируется скриптом:
scripts/generate_datasets.py
Скрипт создает набор текстовых записей о типовых управленческих ситуациях в ресторанном бизнесе. Записи описывают проблемы, показатели и рекомендации, связанные с закупками, запасами, списаниями, поставщиками, food cost, ABC/XYZ-анализом, инвентаризацией и прогнозированием спроса.
Данные не являются реальными операционными данными конкретного ресторана. Они не содержат персональных данных, коммерческой тайны, реальных поставщиков, реальных ресторанов или закрытой финансовой информации.
Файл исходных данных
Исходный файл данных находится по пути:
data/raw/datasets.json
Файл создается командой:
uv run python scripts/generate_datasets.py
Объем данных
Исходный датасет содержит:
1200 текстовых записей
Такой объем выбран для проверки работы RAG-pipeline на корпусе, превышающем минимальный MVP. Корпус из 1200 записей позволяет протестировать ingestion, chunking, indexing и retrieval на более крупном наборе документов.
Структура записи
Каждая запись в data/raw/datasets.json имеет следующую структуру:
{
  "id": 1,
  "name": "Кейс 0001: управление складскими остатками — ресторан авторской кухни",
  "text": "Текстовое описание управленческой ситуации..."
}
Описание полей:
id — уникальный идентификатор записи;
name — название кейса;
text — основной текст документа, который используется для индексации и поиска.
Индексируемые поля
Индексируется только поле:
text
Именно это поле содержит основную смысловую информацию: ресторанный контекст, управленческую проблему, продуктовую категорию, числовые показатели и практическую рекомендацию.
Метаданные
Поля:
id
name
используются как метаданные.
Они нужны для отображения источников в Streamlit-интерфейсе. После обработки данных идентификатор записи используется как doc_id, а название кейса отображается как name.
Тематический состав корпуса
Корпус содержит документы по следующим темам:
•	прогнозирование спроса;
•	управление складскими остатками;
•	оптимизация закупок;
•	контроль списаний;
•	ABC-анализ ингредиентов;
•	XYZ-анализ стабильности потребления;
•	расчет food cost;
•	управление поставщиками;
•	инвентаризация;
•	сезонность спроса;
•	планирование меню;
•	контроль дефицита;
•	аналитика продаж;
•	управление закупочными ценами;
•	анализ оборачиваемости запасов;
•	контроль качества данных;
•	выявление аномалий;
•	планирование банкетов;
•	учет скоропортящихся продуктов;
•	контроль маржинальности блюд.
Примеры вопросов, на которые корпус должен отвечать
Примеры релевантных вопросов:
Как ресторану снизить списания скоропортящихся продуктов?
Какие метрики нужно использовать для контроля закупок в ресторане?
Зачем ресторану нужен ABC-анализ ингредиентов?
Эти вопросы соответствуют тематике корпуса и должны возвращать релевантные фрагменты с источниками.
Negative-вопрос
Пример вопроса вне корпуса:
Кто проживает на дне океана?
Ожидаемое поведение системы: отказ от ответа, потому что в базе знаний нет релевантного контекста по этому вопросу.
Ожидаемый ответ:
В базе не найдено релевантных фрагментов. Ответить по данным невозможно.
Обработка данных
Обработка данных проходит по следующему pipeline:
data/raw/datasets.json
→ data/processed/documents.jsonl
→ data/processed/chunks.jsonl
→ data/index/chunks.jsonl
→ data/index/vectorizer.pkl
→ data/index/matrix.npz
Этапы обработки:
1.	scripts/generate_datasets.py создает исходный файл data/raw/datasets.json.
2.	scripts/ingest.py преобразует записи из JSON в документы.
3.	app/chunker.py нарезает документы на чанки.
4.	scripts/build_index.py строит TF-IDF индекс.
5.	app/retriever.py использует индекс для поиска релевантных фрагментов.
6.	app/generator.py формирует demo-ответ только на основании найденных фрагментов.
7.	app/main.py отображает вопрос, найденные фрагменты, ответ и источники в Streamlit UI.
Файлы, создаваемые после обработки
После запуска сборки индекса создаются или обновляются следующие файлы:
data/processed/documents.jsonl
data/processed/chunks.jsonl
data/index/chunks.jsonl
data/index/matrix.npz
data/index/vectorizer.pkl
Назначение файлов:
documents.jsonl — документы после ingestion;
chunks.jsonl в data/processed — чанки после нарезки;
chunks.jsonl в data/index — чанки, связанные с индексом;
matrix.npz — разреженная TF-IDF матрица;
vectorizer.pkl — сохраненный TF-IDF vectorizer.
Что не индексируется
Не индексируются:
•	служебные файлы проекта;
•	виртуальное окружение .venv;
•	кэш Python __pycache__;
•	кэш тестов .pytest_cache;
•	файлы IDE;
•	временные файлы;
•	README и документация;
•	данные, не относящиеся к корпусу ресторанной аналитики.
Обновление данных
Для обновления исходного датасета нужно выполнить:
uv run python scripts/generate_datasets.py
После обновления данных необходимо пересобрать индекс:
uv run python scripts/build_index.py
После пересборки индекса можно запустить приложение:
uv run streamlit run app/main.py
Или с указанием отдельного порта:
uv run streamlit run app/main.py --server.port 8502
Проверка количества записей
Количество записей в исходном датасете можно проверить командой:
uv run python -c "import json; d=json.load(open('data/raw/datasets.json', encoding='utf-8')); print(len(d['datasets']))"
Ожидаемый результат:
1200
Проверка индекса
После сборки индекса можно проверить содержимое индексированных чанков:
uv run python -c "from pathlib import Path; print(Path('data/index/chunks.jsonl').read_text(encoding='utf-8')[:1000])"
В выводе должны встречаться слова, связанные с тематикой проекта:
ресторан
закупки
списания
запасы
food cost
Ограничения данных
Датасет является учебным и синтетическим. Он предназначен для демонстрации работы RAG-системы, а не для принятия реальных управленческих решений.
Ограничения:
•	данные не отражают фактические продажи, остатки или списания конкретного ресторана;
•	числовые показатели внутри текстов являются учебными;
•	рекомендации носят демонстрационный характер;
•	качество поиска зависит от совпадения терминов в вопросе и корпусе;
•	система использует TF-IDF и cosine similarity, а не внешнюю LLM или embedding-модель.
Принцип работы с нерелевантными вопросами
Система должна отвечать только на основании найденных фрагментов. Если релевантный контекст не найден, она должна вернуть отказ:
В базе не найдено релевантных фрагментов. Ответить по данным невозможно.
Это ограничение нужно для предотвращения нерелевантных или выдуманных ответов.
Критерии готовности данных
Данные считаются подготовленными корректно, если:
•	файл data/raw/datasets.json существует;
•	файл содержит 1200 записей;
•	каждая запись содержит поля id, name, text;
•	поле text содержит содержательное текстовое описание;
•	после сборки индекса создаются chunks.jsonl, matrix.npz и vectorizer.pkl;
•	Streamlit-интерфейс показывает найденные фрагменты с doc_id, score, name и text;
•	negative-вопрос возвращает отказ от ответа.
